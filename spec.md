# Question Bank — Technical Spec

## 1. Goal

Turn a folder of Science Olympiad test PDFs (test + answer key) into a single browsable, topic-organised markdown question bank per event, with diagrams correctly attached to their questions and LLM-validated answers. Optionally, augment the bank with novel questions generated by an LLM from wiki pages or user-supplied reference material. Built so a future randomiser can pull questions from the JSON cache.

Multi-event by design. Events shipped today: `circuit_lab` (Circuit Lab) and `thermodynamics` (Thermodynamics). Adding another is one entry in `events.py` — see §12.

## 2. Architecture

```
scioly.org PDFs ──┐                                       ┌── Markdown
                  ▼                                       │
            <event>/*.pdf                                 │
                  │                                       │
                  │       ┌──────────────────────────┐    │
                  │       │ <event>/.qbank_state.json│ ───┤── <event>/question_bank.md
                  │       └──────────────────────────┘    │
                  │              ▲      ▲                 │
                  ▼              │      │                 │
        [build_question_bank.py] ┘      └─[review_app.py] ┘
                                            ▲   ▲    ▲
        scioly wiki ──┐                     │   │    │
        user PDFs ────┼──> <event>/texts/ ──┘   │    │
                      │                         │    │
                      │                  generate│    │
                      └──────[qgen.py]──────────┘    │
                                                user (browser)
```

All persistent state lives in **one** file per event: `<event>/.qbank_state.json`. Both the CLI pipeline and the Flask UI read and write it; neither overwrites the other.

Per-event configuration (slug, display name, topic taxonomy, keyword weights, filename prefix, wiki page name) lives in `events.py`. `build_question_bank.set_event(slug)` swaps the module's globals (`BASE_DIR`, `IMAGE_DIR`, `OUT_MD`, `STATE_FILE`, `TOPICS`, `TOPIC_KEYWORDS`, `EVENT`) so the rest of the code is event-agnostic.

## 3. Pipeline stages (per PDF)

`process_pair(test_pdf, key_pdf, state, use_vision)` runs the following, with caching at every stage:

1. **Open test PDF** with PyMuPDF (`fitz.open`)
2. **Text extraction** — `page.get_text("text")` for each page. If every page is empty *and* vision is enabled, fall through to OCR (stage 4). Otherwise continue.
3. **Question parsing** (`extract_questions`)
   - Skip cover/instruction pages (detected by markers like "Team Name", "Score:" and fewer than 3 substantive numbered items)
   - Walk pages line-by-line; a line matching `Q_START` (`^(?:question\s+)?(\d{1,3})[\.\)\:\s]`) opens a new question
   - Subsequent non-blank lines accumulate as the question body until the next number or an answer-key marker
   - Duplicate question numbers within one PDF (multi-section tests) get suffixed: `1`, `1b`, `1c`, ...
   - Once an answer-key page is detected, the rest of the PDF is skipped (the key PDF handles answers)
   - `split_choices` splits the body into a `stem` and a `choices[]` list
   - `_strip_points` removes parenthetical point markers like `(2 points)`, `[1 pt]`, `(½ pt each)` from every text field
   - `classify_topic` assigns a topic by keyword scoring (see §6)
4. **Vision OCR** for image-only PDFs (`vision_extract_text_page`) — only when text extraction yielded nothing for the entire PDF. Renders each page at 96 DPI, sends to `claude-haiku-4-5-20251001`, gets back a JSON list of questions.
5. **Image extraction and association** (`associate_images`)
   - Walk every page, extract embedded images via `doc.extract_image(xref)` (skip <2 KB — they're logos and bullets)
   - Save under `<event>/images/<topic_slug>_<src>_p<n>_i<idx>_<hash>.<ext>`
   - For each page with images, ask Haiku vision: *"Which question (or PREV) does each figure illustrate?"* — pass the rendered page + the question numbers visible on it
   - Fall back to y-coordinate heuristic (image attaches to the last question whose text started above it) when vision is unavailable or fails
6. **Answer extraction**
   - Open key PDF, run `extract_answers` page-by-page
   - A page must look like an answer key (keyword OR ≥5 short numbered lines) to be considered
   - Answers >180 chars or matching `Q_START` are rejected (they're reprinted questions, not answers)
   - If the key is image-only, OCR each page via `vision_extract_key_page`
   - For multi-section tests with duplicate numbers, text-based answer matching is skipped entirely — empty beats wrong
7. **Apply annotations** (`apply_annotations`) — layer user edits from `state.annotations[<pdf>]` on top: deletions, additions, field overrides, image reassignments, validation verdicts. See §5.
8. **Persist** — write `state.questions[<pdf>]` and `state.vision[<pdf>]` to JSON.

## 4. State file (`.qbank_state.json`)

```jsonc
{
  "questions": {
    "<pdf_name>.pdf":  [ Question, ... ],
    "_generated_<event>.pdf": [ Question, ... ]   // LLM-generated, marked via validation.generated
  },
  "vision": {
    "<pdf_name>.pdf": {              // cached Haiku responses, keyed by use-case
      "p<N>":         { "0": "5", "1": "PREV" },     // image-to-question per page
      "ocr_p<N>":     [ { "number": "1", "text": "..." } ],   // text OCR
      "key_ocr_p<N>": { "1": "B", "2": "14 Ω" }     // key OCR
    }
  },
  "manual": { "<pdf>": { "edited_at": "ISO-8601" } },
  "annotations": {
    "<pdf>": Annotations
  }
}
```

Generated questions live under the synthetic key `_generated_<event>.pdf` so they appear alongside pipeline-extracted ones in the markdown but are easy to identify and remain immune to reprocess (which only touches real PDFs).

### Question

```jsonc
{
  "number":   "5",                  // or "5b", "5c" for multi-section duplicates
  "topic":    "Series & Parallel Circuits",
  "focus":    "",                   // optional: rotating sub-topic (e.g. "Endocrine"); see §6b
  "text":     "What is ... ?",      // the stem; MC options live in `choices`
  "choices":  [ { "letter": "A", "text": "..." }, ... ],
  "answer":   "10V",
  "images":   [ "filename.png", ... ],
  "source":   "2020 Div-C: duke",
  "year":     "2020",
  "division": "C",
  "page":     3,                    // for the review UI
  "validation": Validation          // optional, set by LLM check
}
```

### Annotations

User edits, replayed on top of pipeline output every time `process_pair` runs. Survive `--rebuild`, the CLI re-run, and the UI's *Reprocess* button.

```jsonc
{
  "field_overrides": {
    "5": { "text": "...", "choices": [...], "answer": "...", "topic": "..." }
  },
  "added": [ Question, ... ],          // user-created questions
  "deleted": [ "12", "13" ],           // qnums to drop
  "image_overrides": {
    "assignments": { "fname.png": "5" },
    "detached":    [ "other.png" ]
  },
  "regions": [                          // provenance only — drag-capture history
    { "number": "5", "field": "stem", "page": 3, "target": "test",
      "x": 50, "y": 80, "w": 600, "h": 120, "dpi": 120,
      "captured_at": "ISO-8601" }
  ],
  "validations": { "<qnum>": Validation }
}
```

### Validation

```jsonc
{
  "status":         "correct" | "incorrect" | "uncertain" | "unavailable",
  "correct_answer": "10V" | null,
  "rationale":      "Ohm's law gives V=IR=2×5=10V.",
  "source":         "Ohm's Law (V=IR)",
  "validated_at":   "ISO-8601",
  "model":          "claude-haiku-4-5-20251001",
  "text_at_validation":   "... first 300 chars of question text ...",
  "answer_at_validation": "10V",
  "stale":          false              // set true if text/answer changed since
}
```

## 5. Annotation replay (`apply_annotations`)

Order matters:

1. Drop questions whose number appears in `deleted`
2. Append `added` questions (skip if number collision)
3. Apply `field_overrides[qnum]` to all questions (pipeline-extracted *and* added)
4. Image overrides: strip `detached` and any reassigned images from every question, then attach `assignments[fname] = qnum` to the target
5. Attach `validations[qnum]` to each question; mark `stale: true` if `text_at_validation` or `answer_at_validation` no longer matches

A reprocess thus preserves user effort while still applying new pipeline logic to fields the user never touched.

## 6. Topic taxonomy

Per-event. Each event in `events.py` declares a tuple of `topics` (always ending in `Other / General`) and a `topic_keywords` dict of `{topic: [(phrase, weight), ...]}`. `classify_topic(text)` lowercases the text, sums weights for every phrase that appears, and picks the highest-scoring topic; ties go to insertion order; an empty result falls through to *Other / General*.

Tuning principles that apply to every event:
- **Don't include generic terms** that appear across many topics. For Circuit Lab, that means omitting "voltage"/"current" from *Basic Electrical Concepts*; for Thermodynamics, omitting "heat"/"energy" from *Temperature & Heat*. Otherwise one bucket swallows everything.
- **Use multi-word phrases** when possible (`"ohm's law"`, `"q = mc"`, `"carnot cycle"`) — they're more discriminative than single tokens.
- **Weight 5** for unambiguous names of the topic itself (`"kirchhoff"`, `"carnot"`), **4** for strong-signal multi-word phrases, **3** for moderately specific terms, **2** for weak/common-but-relevant terms.

Custom topics created in the UI (via the "+ New topic..." dropdown option) are stored as `q.topic` strings and surface in future dropdowns without modifying `events.py`.

Current taxonomies (full lists live in `events.py`):

- **Circuit Lab** — 10 buckets: Basic Electrical Concepts · Series & Parallel · Kirchhoff's Laws · Power & Energy · Capacitors · Inductors & Magnetism · AC Circuits & Phasors · Semiconductors & Diodes · Measurement & Lab Skills · Circuit Diagrams & Symbols
- **Thermodynamics** — 11 buckets: Temperature & Heat · Specific Heat & Calorimetry · Thermal Expansion · Heat Transfer · Phase Changes & Latent Heat · Laws of Thermodynamics · Gas Laws & Ideal Gases · Entropy & Free Energy · Heat Engines & Refrigerators · Insulation & Building Science · Lab Skills & Measurement

## 6b. Rotating foci

Some Sci-Oly events change their sub-topic focus each year (Anatomy & Physiology cycles through body systems, Dynamic Planet alternates oceanography/glaciers/tectonics, Water Quality picks freshwater or saltwater, Materials Science assigns a single material family). The pipeline models this with two pieces of state:

- `Event.foci: tuple[str, ...]` — the rotation set declared at registration time. Empty for events without rotation (Circuit Lab, Thermodynamics).
- `Question.focus: str` — which rotation slot this question belongs to. Empty for non-rotating events, or for questions the user hasn't tagged yet.

**Auto-population paths:**

- **scio.ly scrape** uses scio.ly's `"<Event> - <Focus>"` naming convention. When the UI's focus dropdown is set, the scrape rewrites `event_name` to `"Anatomy - Endocrine"` before hitting the API AND passes `focus="Endocrine"` to the scraper so every normalized question gets tagged. `scrape_scioly.split_event_and_focus()` also auto-derives `focus` from any event name that already has the `" - "` suffix, so the same code path handles a directly-typed `"Anatomy - Endocrine"`.
- **PDF extraction** does not auto-tag focus — the year-to-focus mapping isn't encoded in scioly_tests.json and varies by division. Users tag focus per-question via the review UI's focus dropdown; the edit lands in `annotations.field_overrides` and replays on reprocess.

**Markdown rendering** (`build_markdown`):

- If `EVENT.foci` is empty OR no question has a focus set → flat layout: `# Event` → `## Topic` → `### Q*`.
- Else: nested layout: `# Event` → `# Focus` (top-level for jump-from-TOC) → `## Topic` → `### Q*`. Foci are emitted in the order declared on the event; questions with `focus == ""` go into a synthetic `(no focus)` group at the bottom. Question numbering is contiguous across the whole document.

**UI:**

- Sources page shows a Focus dropdown on the scrape form only when `EVENT.foci` is non-empty.
- Review cards show a Focus dropdown only when `STATE.foci` is non-empty or the question already has a focus value. The dropdown has a `(no focus)` option and a `+ New focus…` option that prompts for a new value and adds it to `extraFoci` for the rest of the session.
- The new-event form has a "Rotating foci" input that accepts comma-separated values; persisted to `events_custom.json`.

## 7. Multiple-choice splitting (`split_choices`)

Detects MC options in three formats: `a. text`, `A) text`, `(a) text`. Returns `(stem, choices[])` or `(text, [])` if it doesn't look like MC.

Guards (the regex finds too many false positives without these):

- Letters must be strictly ascending starting from A (with one exception below)
- **Implicit-A inference**: if only `b. ... d. ...` survived OCR, the stem ends at the last `?` or `____` before the first letter, and the implicit A is whatever's between
- Average option length must be < 80 chars (longer means it's a multi-part question with `a) ... b) ...`, not MC)
- Stem must contain an interrogative cue (`?`, `____`, `:`, or a word like *which/what/who/how/calculate*) — without this, what looks like the stem is probably the body of an option whose marker was lost

## 8. Markdown output

`build_markdown(all_questions)` groups by topic in the order listed in §6. Each question renders as:

```
### Q<n>
*<source> · #<number>*

<stem>

- **A.** ...
- **B.** ...

![Figure](images/...)

**Answer:** ...

> ✓ **Verified: Correct**  <rationale>  *Source: <source>*
```

Math markers (`$...$`, `$$...$$`) pass through verbatim — GitHub-flavored markdown renders them. Validation blockquote omitted when no verdict exists.

## 9. Flask review UI

State store and pipeline are the same. The UI is just a graphical front-end that produces annotations and generated questions.

### Pages

- `GET /` — event landing page (one row per registered event)
- `GET /event/<slug>/` — sortable PDF list for one event. Includes an upload-test-PDF form (test + optional key).
- `GET /event/<slug>/review/<pdf>` — single-PDF review (PDF page + question cards). Accepts `?q=<number>` to focus a specific question on load (used by the "Open source ↗" link from the browse page).
- `GET /event/<slug>/sources` — source-material management + LLM question generation
- `GET /event/<slug>/browse` — event-wide question browser: every question across every bucket on one filterable/sortable page. Filters: full-text search, topic, focus, source, validation status, MCQ-vs-FRQ, has-image. Sorts: Q#, topic, focus, source, validation, length, bucket-then-Q#. Each card links back to the source PDF's review page with that Q pre-focused.

Every API route is namespaced under `/event/<slug>/`. Each handler calls `_select_event(slug)` first, which mutates the `build_question_bank` module globals (`BASE_DIR`, `IMAGE_DIR`, `OUT_MD`, `STATE_FILE`, `TOPICS`, `EVENT`) so the rest of the request runs in the right context. Flask's dev server is single-threaded by default, so this works correctly without locking.

### Region-coordinate convention

The frontend sends a region in **natural image pixels** at the requested DPI (default 120). The backend converts to PDF points:

```python
f = 72.0 / dpi
pdf_rect = fitz.Rect(x*f, y*f, (x+w)*f, (y+h)*f)
text = page.get_text("text", clip=pdf_rect)
```

Math extraction re-renders that same `pdf_rect` at 240 DPI for vision clarity before sending to Haiku.

### Question bounding boxes (`/qboxes`)

Derived on-demand from PyMuPDF text blocks rather than stored at extraction time, so this works on any PDF in the bank without rerunning the pipeline.

Algorithm:
1. `page.get_text("blocks")` returns `(x0, y0, x1, y1, text, block_no, type)` tuples.
2. Filter to text blocks (type 0) and sort by `y0`.
3. For each block, inspect the first non-blank line — if `Q_START` matches, record an anchor `(y0, qnum, block)`.
4. For each consecutive pair of anchors, compute the bbox spanning from `anchor[i].y0` to `anchor[i+1].y0` (or page height for the last one), widened to the leftmost/rightmost edges of every text block inside that vertical slice.
5. Return PDF-point coords; the frontend scales to displayed image pixels.

UI behavior:
- Boxes render as soft-green overlays after the page image loads.
- **Focused** (the q whose `_id` matches `focusedId`): orange box.
- **Unmatched** (the box's qnum doesn't appear in `STATE.questions`): dashed red box — this is the "the algorithm thinks there's a Q here but the parser dropped it" diagnostic.
- Click a box → jumps to that question's card and focuses it.
- Toggle on/off via the ▢ button in the zoom controls; preference persists in `localStorage` (default ON).
- Key PDF mode hides bboxes (key PDFs aren't question-numbered the same way).

### Status-bar lifecycle

`setStatus(html, kind)` writes into the header's status span and schedules a clear after 4.5s. Earlier versions cleared only when `el.innerHTML === html`, but **browsers normalise attribute quoting when reading back `innerHTML`** (e.g. `'spinner'` → `"spinner"`), so any status containing inline HTML failed the equality check and the message — and any embedded spinner — never cleared. Current implementation uses a monotonically incremented `_statusSeq` token: each call captures its sequence, and the timeout only clears when no newer call has run since. This means the spinner stops as soon as the work finishes, regardless of what shape the status string takes.

Handler convention: every async UI handler that shows a "Working…" status must also write a definitive terminal status — success, cancel, no-op, or error — on every code path, so the user always knows the operation ended. Disable the triggering button via `try/finally` to prevent re-entry while a long-running call is in flight.

### Add-question-from-region UX

When the user clicks **+ Add question from region** and drags a rectangle, the frontend:

1. Sends the region to `extract-region` with `parse_choices: true`
2. Receives `{text, stem, choices}`
3. Auto-assigns the next available numeric Q# — `max(existing.number) + 1` — **no prompt**
4. If `choices` is non-empty, uses `stem` for the question text and populates the choices list; otherwise uses `text` as the stem with an empty choices list

This means dragging a region that covers a stem plus its `A. ... B. ... C. ...` options produces a fully-structured question card in one drag.

### Region-coordinate convention

The frontend sends `(x, y, w, h)` in **natural image pixels** at the requested DPI. The backend converts:

```python
f = 72.0 / dpi
pdf_rect = fitz.Rect(x*f, y*f, (x+w)*f, (y+h)*f)
text = page.get_text("text", clip=pdf_rect)
```

Math extraction re-renders the same `pdf_rect` at 240 DPI for clarity before sending to vision.

## 10. Vision prompts

Stable prompts kept in `build_question_bank.py`:

- **Image-to-question association** — gives Haiku the rendered page, the question numbers that *start* on it, and the count of figures. Asks for `{"<fig_idx>": "<qnum>" | "PREV"}`. The `PREV` escape hatch covers figures that illustrate a question started on the previous page.
- **Page OCR** — for fully image-based PDFs. Asks for `{"questions":[{"number","text","has_figure"}], "answers":[...]}`. Empty `{}` for cover/instruction pages.
- **Key OCR** — for image-based key PDFs. Asks for `{"1":"B","2":"14 Ω", ...}`.
- **Math → LaTeX** — sends a high-DPI crop. Asks for the LaTeX with no delimiters; reply `NONE` if no math.
- **Answer validation** — sends question + choices + recorded answer. Asks for a JSON verdict with status, correct answer, rationale, primary source.

All vision calls go through `_vision_call(b64, prompt, max_tokens)` and are cacheable: callers store results in `state.vision[<pdf>][<key>]` and check the cache before paying again.

## 11. Costs

Haiku 4.5 pricing roughly 1¢ per ≈100 vision calls. Numbers below are for the Circuit Lab corpus (22 PDFs, ~800 questions, ~340 images). Other events scale linearly with PDF count, page count, and question count.

| Operation | Calls | Approx cost |
|---|---|---|
| Image-to-question association | once per page with images (~100) | $0.30–0.50 |
| OCR for image-only PDFs | once per page (~30 pages across 2 PDFs) | $0.10 |
| Math capture | per equation, opt-in | $0.005 each |
| Answer validation | per question, opt-in | $0.003 each |
| Question generation | per Generate-button click (~5 candidates) | $0.01 each |

Bulk-validating the whole 800-question bank ≈ $2.40. The UI shows a per-page cost estimate before running.

## 11b. scioly.org bot bypass (`download_event.py`)

scioly.org uses **Anubis** JavaScript proof-of-work bot protection. Plain `requests` always gets back HTML challenge pages.

The downloader handles this transparently:

1. Issues GETs with a Chrome User-Agent and any cookies from `.scioly_cookies.json`.
2. If a response is `text/html` containing "not a bot" or "anubis", that's the challenge.
3. **First time only** in the run: launches headed Chromium via Playwright, navigates to `https://scioly.org/tests/`, waits up to 60s for `document.title` to no longer contain "not a bot" (signals PoW completion), grabs `context.cookies()`, and saves them to `.scioly_cookies.json`.
4. Re-applies the cookies to the requests session and retries the same URL.

Subsequent runs reuse the saved cookies until they expire. `--reauth` discards them and re-runs the bypass. `--no-bypass-bot` skips Playwright entirely and just fails on a challenge.

Prerequisites: `pip install playwright` and `playwright install chromium` once.

Detection helper: `_looks_like_challenge(response)` checks `content-type` + a substring scan of the first 600 chars.

## 11c. Source materials (`texts.py`)

Each event has a `<event>/texts/` directory holding reference material the LLM can draw on. Three file kinds:

- `wiki.md` — fetched from `event.wiki_url` (scioly.org/wiki/index.php/<Page>)
- `<name>.pdf` — user-supplied PDF (textbook chapter, study guide, notes)
- `<name>.md` — derived markdown (via `pdf_to_markdown`) **or** standalone reference text

`list_sources(event)` returns one entry per source. A PDF + sibling `.md` are merged into one row; the markdown is shown as derived metadata.

`scrape_wiki(event, cookies)`:
- Builds the URL from `event.wiki_url` (default: `name.replace(' ', '_')`; can be overridden via `Event.wiki_page`)
- Uses the saved Anubis cookies if provided
- Extracts `div.mw-parser-output` with BeautifulSoup, strips `mw-editsection`/`noprint`/`toc`/navbox noise
- Converts to markdown via `markdownify(heading_style="ATX")`
- Saves with a small `# <event> — Wiki` header to `<event>/texts/wiki.md`

`pdf_to_markdown(pdf_path)`:
- PyMuPDF text-only extraction, page-by-page
- Writes `## Page N\n\n<text>` blocks to a sibling `.md`

`read_source_text(event, name)`:
- Resolves `foo.pdf` → `foo.md` (errors if not yet processed)
- Resolves `bar.md` → `bar.md`

### Shared textbooks (`textbooks/`, top level — not under any one event)

Textbooks differ from per-event sources in two ways: they're uploaded once and usable from every event, and they're split by chapter so generation targets one chapter's pages rather than the whole book. Storage layout: `textbooks/<id>.pdf` + `<id>.md` (built via the unmodified `pdf_to_markdown` above) + `<id>.chapters.json`.

`detect_chapters(pdf_path, md_text)` — two strategies tried in order:
1. The PDF's own embedded outline (`fitz.open(pdf_path).get_toc()`). When present, the top-level entries are taken as-is (`{title, start_page: page}`) — most reliable when it exists.
2. A regex scan of `md_text` for heading lines matching `^(?:chapter|unit|section)\s+(\d{1,3}|[ivxlcdm]+|[a-z]+)\b` (case-insensitive, start-of-line only so mid-sentence mentions don't match), tracking the most recent `## Page N` marker `pdf_to_markdown` already emits to assign each match's `start_page`.

If neither strategy finds anything, returns `{chapters: [], source: "manual", needs_manual_chapters: true}` so the caller can fall back to manual entry. Otherwise each chapter's `end_page` is filled in from the next chapter's `start_page - 1` (last chapter runs to the PDF's final page).

The manual-entry route (`POST /api/textbooks/<id>/chapters`) applies the identical cascading-`end_page` rule when a chapter's `end_page` is omitted, so the two code paths produce consistent boundaries.

## 11d.1. scio.ly scraper (`scrape_scioly.py`)

scio.ly exposes a public, no-auth REST endpoint that returns Science Olympiad practice questions:

```
GET https://www.scio.ly/api/questions?event=<Display>&question_type=mcq|frq&limit=N&division=B|C
```

Returns `{"success": true, "data": [{ id, question, options, answers, subtopics, tournament, division, difficulty, base52 }, ...]}`. `answers` is a list of indices into `options` for MCQ; for FRQ it holds the canonical text.

The module exposes:
- `fetch_batch(event_name, question_type, limit, division)` — one HTTP call, returns raw records
- `_normalize(raw)` — converts to canonical Question shape: maps `answers: [idx]` → letter "A"/"B"/…; resolves `subtopics[0]` to the event's topic taxonomy when it matches, else falls back to `classify_topic`; sets `source: "scio.ly · <tournament>"`; carries through `_scioly_id` and `_scioly_difficulty` as provenance
- `scrape_questions(event_name, count, types, division, existing_scioly_ids)` — round-robins over types, rate-limits at 0.5 s between API calls (`REQUEST_DELAY`), dedups against the provided UUID set

**Things scio.ly questions notably lack:** images (the `images` list is always `[]`), and many questions are partial — context-stripped or with answers that depend on missing background. The Haiku validator catches these as `uncertain`/`incorrect` so the user can drop them in bulk via the UI's quick-filter buttons.

**Two-layer dedup**:

1. **Exact (UUID)** — each scraped question's UUID is stored in `q._scioly_id`. The scrape endpoint collects every UUID already in `_scioly_<slug>.pdf` and asks the scraper to skip them — so hitting Scrape repeatedly walks deeper into scio.ly's archive instead of returning the same top results. Counted under `skipped_id_dups`.
2. **Fuzzy (text)** — the endpoint also gathers every question from every bucket (PDF-extracted, LLM-generated, prior scio.ly scrapes) into a single `existing_questions` list and passes it to `scrape_questions(..., existing_questions=...)`. The scraper calls `qgen.is_duplicate(candidate, existing_questions + already_kept_in_this_batch, threshold=0.4)` on each normalized question. Hits are dropped into `rejected_text_dups[]` with `{text, matched_q_number, matched_q_source, matched_q_text, _scioly_id}` so the UI can render the side-by-side comparison.

To make the fuzzy layer work despite scio.ly returning random samples per call, the scraper requests `min(per_type * 3 + 10, 50)` records per type — over-fetching by ~3× so dedup filtering still leaves enough to satisfy `count`.

## 11d. Question generation (`qgen.py`)

`generate_questions(source_text, n, types, existing_questions, max_chunks=1)` returns:

```jsonc
{
  "candidates":     [ Candidate, ... ],
  "rejected":       [ {"reason": "duplicate" | "too short", "matched": "<qnum>" | null, "text": "..."} ],
  "warnings":       [ "chunk 1: ...", ... ],     // non-fatal issues per chunk
  "model":          "claude-haiku-4-5-20251001",
  "chunks_used":    1,
  "raw_count":      5,
  "out_max_tokens": 8850,                          // budget actually requested
  "generated_at":   "ISO-8601",
  // If the LLM fails outright, these appear instead of candidates/rejected:
  "error":          "LLM response was not valid JSON.",
  "stop_reason":    "max_tokens",                  // from Anthropic API
  "raw_response":   "<first 1500 chars of raw LLM output>"
}
```

**Token budgeting.** `out_max_tokens = min(16000, 600 + 550 * n * chunks)`. Each generated question with rationale + source snippet runs ~500 tokens; a few-hundred-token overhead covers the JSON envelope. Without this scaling, large `n` requests would silently get truncated mid-JSON, producing 0 candidates with no error.

**Failure surfacing.** When `_parse_json` returns None on the first (or only) chunk, the function returns `error` + `stop_reason` + the first 1500 chars of `raw_response` so the UI can show the user what came back. Per-chunk failures on a multi-chunk run accumulate into `warnings` and the run continues.

**Frontend UX during generation.** The Generate button is disabled and replaced with "Generating…" while in-flight; a progress panel shows the elapsed-time counter with phase hints ("Sending request…", "Waiting for Haiku…", "Still drafting…", "Taking longer than usual — you can Cancel if needed"). A Cancel button uses `AbortController` to abort the fetch. HTTP errors and non-JSON server responses are caught and displayed in the status bar.

Pipeline:

1. **Chunk** source via `chunk_text(text, max_chars=9000)` on paragraph boundaries.
2. For each chunk (up to `max_chunks`, default 1), build a structured prompt:
   - Identifies the event by name
   - Spells out per-type expectations (mc: 4 choices A-D with plausible distractors; short: 1-2 sentence answers; numerical: answers with units + equation in rationale)
   - Lists the event's `topics` and asks Haiku to pick one per question
   - Shows up to 30 existing question stems as "do not duplicate"
   - Demands JSON-only output with schema: `{questions:[{type, topic, text, choices, answer, rationale, source_snippet}]}`
3. **Clean** each candidate: `_strip_points` on text/answer/choices, re-letter choices so they start at A, classify topic from text if Haiku picked something off-list.
4. **Dedup**: `is_duplicate(cand, existing, threshold=0.4)` uses Jaccard similarity on word-3-grams of the stem. Auto-rejects with `reason: "duplicate"` + the matched question number. Candidates are also deduped against earlier candidates in the same batch.

`candidate_to_question(cand, number, source_label)` converts a kept candidate to the canonical Question shape used by the rest of the bank, populating the `validation` field with the LLM rationale, source snippet, and `generated: true` so generated questions are distinguishable in the markdown and UI. Generated questions are stored under a synthetic state key `_generated_<event>.pdf` so they survive reprocess and show up alongside real ones.

## 12. Adding another event

Two paths, depending on whether you need topic-keyword classification from day one.

### 12a. Hard-coded events (`events.py`)

The recommended path when you have a topic taxonomy worked out:

```python
EVENTS["my_event"] = Event(
    slug="my_event",                       # → directory my_event/ and URL /event/my_event/
    name="My Event",                       # display name + vision-prompt context
    event_match=("my event",),             # scioly_tests.json event-name substrings
    filename_prefix="myevent",             # PDF filename prefix (matches scioly URLs)
    topics=("Topic A", "Topic B", "Other / General"),
    topic_keywords={"Topic A": [("kw", 4)], ...},
)
```

### 12b. UI-registered events (`events_custom.json`)

For quick experimentation or events whose taxonomy isn't fully defined yet:

- **Endpoint**: `POST /api/events` (top-level, no event slug). Body: `{slug, name, filename_prefix, event_match, wiki_page, topics}`. Strings for `event_match` and `topics` are accepted as either JSON arrays or comma-separated strings.
- **Validation**: slug must match `^[a-z][a-z0-9_]*$`; name and filename_prefix are required; filename_prefix must have no spaces or slashes.
- **Persistence**: written to `events_custom.json` at the repo root via `add_custom_event()`. Loaded on import in `events.py`.
- **Behavior**: at module import time `events.py` builds the hard-coded `EVENTS` dict, then `_load_custom_events()` merges in entries from `events_custom.json` (skipping any slug that collides with a built-in).
- **`topic_keywords` is empty** for UI-registered events, so `classify_topic` will fall through to "Other / General" for every pipeline-extracted question. The user re-topics from the dropdown on each question card. `qgen` generation is unaffected — it asks Haiku to pick a topic.
- **Removal**: `remove_custom_event(slug)` deletes from the dict and re-saves. Built-ins (`circuit_lab`, `thermodynamics`) cannot be removed; checked via `is_builtin(slug)`.

### Common to both paths

The CLI tools (`download_event.py`, `build_question_bank.py`) take `--event <slug>` and look the rest up from the registry. The review UI auto-discovers events on the landing page; routes are namespaced under `/event/<slug>/`. Either way an `<event>/` directory is auto-created on first access.

`scioly_tests.json` already contains metadata for **every event** scraped from scioly.org (887 entries across all years), so new-event downloads need no additional scraping (only Anubis bypass — see §11b).

### `events_custom.json` schema

```jsonc
{
  "events": [
    {
      "slug":            "fermi_questions",
      "name":            "Fermi Questions",
      "event_match":     ["fermi"],            // [] is allowed for upload-only events
      "filename_prefix": "fermiquestions",
      "topics":          ["Estimation", "Orders of magnitude", "Other / General"],
      "topic_keywords":  {},                   // always empty when created via UI
      "wiki_page":       "Fermi_Questions"     // "" → derived from `name`
    }
  ]
}
```

## 13. Known limitations

- **Multi-section tests** lose most automatic answer matching (e.g. tests with separate "History", "Capacitors", "Op-Amps" sections each restarting at 1). The pipeline detects this case (any suffixed question number) and skips text-based key matching to avoid wrong answers. The reviewer is the recommended path for these.
- **OCR-only PDFs** with no embedded text rely entirely on Haiku; quality varies. Three known PDFs (`cookiepie1`, `ivanioc`, plus one mis-encoded filename) still yield 0 questions because they're structurally empty.
- **Topic classification is keyword-based**. It works well on the vocabulary captured in each event's `topic_keywords`; novel phrasings fall to "Other / General". Custom topics from the UI live on individual questions but don't update the keyword scorer — so the next pipeline run on a fresh PDF won't auto-classify into a user-invented topic.
- **Validation accuracy is bounded by Haiku 4.5**. Treat verdicts as suggestions for a human to confirm, not gospel.
- **scioly.org bot protection** is Anubis JavaScript proof-of-work. `download_event.py` and the wiki scraper auto-handle it via Playwright (see §11b). Requires `playwright install chromium` once. Cookies expire after a few hours; `--reauth` (downloader) or simply re-running scrape-wiki after expiry forces a fresh challenge.
- **Generated questions are not validated against authoritative sources** — Haiku's rationale + source snippet are the LLM's claim, not a check. Treat them as draft material: use the answer-validation button (§9) to cross-check generated questions before relying on them.
- **Generation prompts only use the first chunk** of a source by default (`max_chunks=1`). For a long textbook this skews coverage to the early sections. Repeat the Generate action or pre-chunk the source by hand to spread coverage.
