# Question Bank — Technical Spec

## 1. Goal

Turn a folder of Science Olympiad test PDFs (test + answer key) into a single browsable, topic-organised markdown question bank per event, with diagrams correctly attached to their questions and LLM-validated answers. Optionally, augment the bank with novel questions generated by an LLM from wiki pages or user-supplied reference material. Built so a future randomiser can pull questions from the JSON cache.

Multi-event by design. Events shipped today: `circuit_lab` (Circuit Lab) and `thermodynamics` (Thermodynamics). Adding another is one entry in `events.py` — see §12.

## 2. Architecture

```
scioly.org PDFs ──┐                                       ┌── Markdown
                  ▼                                       │
            <event>/*.pdf                                 │
                  │                                       │
                  │       ┌──────────────────────────┐    │
                  │       │ <event>/.qbank_state.json│ ───┤── <event>/question_bank.md
                  │       └──────────────────────────┘    │
                  │              ▲      ▲                 │
                  ▼              │      │                 │
        [build_question_bank.py] ┘      └─[review_app.py] ┘
                                            ▲   ▲    ▲
        scioly wiki ──┐                     │   │    │
        user PDFs ────┼──> <event>/texts/ ──┘   │    │
                      │                         │    │
                      │                  generate│    │
                      └──────[qgen.py]──────────┘    │
                                                user (browser)
```

All persistent state lives in **one** file per event: `<event>/.qbank_state.json`. Both the CLI pipeline and the Flask UI read and write it; neither overwrites the other.

Per-event configuration (slug, display name, topic taxonomy, keyword weights, filename prefix, wiki page name) lives in `events.py`. `build_question_bank.set_event(slug)` swaps the module's globals (`BASE_DIR`, `IMAGE_DIR`, `OUT_MD`, `STATE_FILE`, `TOPICS`, `TOPIC_KEYWORDS`, `EVENT`) so the rest of the code is event-agnostic.

## 3. Pipeline stages (per PDF)

`process_pair(test_pdf, key_pdf, state, use_vision)` runs the following, with caching at every stage:

1. **Open test PDF** with PyMuPDF (`fitz.open`)
2. **Text extraction** — `page.get_text("text")` for each page. If every page is empty *and* vision is enabled, fall through to OCR (stage 4). Otherwise continue.
3. **Question parsing** (`extract_questions`)
   - Skip cover/instruction pages (detected by markers like "Team Name", "Score:" and fewer than 3 substantive numbered items)
   - Walk pages line-by-line; a line matching `Q_START` (`^(?:question\s+)?(\d{1,3})[\.\)\:\s]`) opens a new question
   - Subsequent non-blank lines accumulate as the question body until the next number or an answer-key marker
   - Duplicate question numbers within one PDF (multi-section tests) get suffixed: `1`, `1b`, `1c`, ...
   - Once an answer-key page is detected, the rest of the PDF is skipped (the key PDF handles answers)
   - `split_choices` splits the body into a `stem` and a `choices[]` list
   - `_strip_points` removes parenthetical point markers like `(2 points)`, `[1 pt]`, `(½ pt each)` from every text field
   - `classify_topic` assigns a topic by keyword scoring (see §6)
4. **Vision OCR** for image-only PDFs (`vision_extract_text_page`) — only when text extraction yielded nothing for the entire PDF. Renders each page at 96 DPI, sends to `claude-haiku-4-5-20251001`, gets back a JSON list of questions.
5. **Image extraction and association** (`associate_images`)
   - Walk every page, extract embedded images via `doc.extract_image(xref)` (skip <2 KB — they're logos and bullets)
   - Save under `<event>/images/<topic_slug>_<src>_p<n>_i<idx>_<hash>.<ext>`
   - For each page with images, ask Haiku vision: *"Which question (or PREV) does each figure illustrate?"* — pass the rendered page + the question numbers visible on it
   - Fall back to y-coordinate heuristic (image attaches to the last question whose text started above it) when vision is unavailable or fails
6. **Answer extraction**
   - Open key PDF, run `extract_answers` page-by-page
   - A page must look like an answer key (keyword OR ≥5 short numbered lines) to be considered
   - Answers >180 chars or matching `Q_START` are rejected (they're reprinted questions, not answers)
   - If the key is image-only, OCR each page via `vision_extract_key_page`
   - For multi-section tests with duplicate numbers, text-based answer matching is skipped entirely — empty beats wrong
7. **Apply annotations** (`apply_annotations`) — layer user edits from `state.annotations[<pdf>]` on top: deletions, additions, field overrides, image reassignments, validation verdicts. See §5.
8. **Persist** — write `state.questions[<pdf>]` and `state.vision[<pdf>]` to JSON.

## 4. State file (`.qbank_state.json`)

```jsonc
{
  "questions": {
    "<pdf_name>.pdf":  [ Question, ... ],
    "_generated_<event>.pdf": [ Question, ... ]   // LLM-generated, marked via validation.generated
  },
  "vision": {
    "<pdf_name>.pdf": {              // cached Haiku responses, keyed by use-case
      "p<N>":         { "0": "5", "1": "PREV" },     // image-to-question per page
      "ocr_p<N>":     [ { "number": "1", "text": "..." } ],   // text OCR
      "key_ocr_p<N>": { "1": "B", "2": "14 Ω" }     // key OCR
    }
  },
  "manual": { "<pdf>": { "edited_at": "ISO-8601" } },
  "annotations": {
    "<pdf>": Annotations
  }
}
```

Generated questions live under the synthetic key `_generated_<event>.pdf` so they appear alongside pipeline-extracted ones in the markdown but are easy to identify and remain immune to reprocess (which only touches real PDFs).

### Question

```jsonc
{
  "number":   "5",                  // or "5b", "5c" for multi-section duplicates
  "topic":    "Series & Parallel Circuits",
  "focus":    "",                   // optional: rotating sub-topic (e.g. "Endocrine"); see §6b
  "text":     "What is ... ?",      // the stem; MC options live in `choices`
  "choices":  [ { "letter": "A", "text": "..." }, ... ],
  "answer":   "10V",
  "images":   [ "filename.png", ... ],
  "source":   "2020 Div-C: duke",
  "year":     "2020",
  "division": "C",
  "page":     3,                    // for the review UI
  "context_id": "ctx_1",            // optional: links to a shared Context block (case-study passage/table/diagram) — see Context below
  "validation": Validation,         // optional, set by either an LLM check or a human (see Validation below)
  "lastEditedBy":       "srikanth", // username of whoever last edited this question (server-stamped from the session, never client-supplied)
  "lastEditedDateTime": "2026-06-21T13:04:47"  // ISO-8601, set alongside lastEditedBy
}
```

`lastEditedBy`/`lastEditedDateTime` are set by `api_patch_question` (`review_app.py`) on every edit made through the Browse page's always-editable cards — including a validation change, since setting/overriding a verdict is itself an edit. Pre-existing questions (from before these fields existed) were backfilled once via `backfill_last_edited.py` (idempotent, safe to re-run) with `lastEditedBy="srikanth"` and `lastEditedDateTime` taken from that bucket's existing `manual[bucket].edited_at` where available.

### Annotations

User edits, replayed on top of pipeline output every time `process_pair` runs. Survive `--rebuild`, the CLI re-run, and the UI's *Reprocess* button.

```jsonc
{
  "field_overrides": {
    "5": { "text": "...", "choices": [...], "answer": "...", "topic": "...",
           "validation": Validation, "lastEditedBy": "...", "lastEditedDateTime": "..." }
  },
  "added": [ Question, ... ],          // user-created questions
  "deleted": [ "12", "13" ],           // qnums to drop
  "image_overrides": {
    "assignments": { "fname.png": "5" },
    "detached":    [ "other.png" ]
  },
  "regions": [                          // provenance only — drag-capture history
    { "number": "5", "field": "stem", "page": 3, "target": "test",
      "x": 50, "y": 80, "w": 600, "h": 120, "dpi": 120,
      "captured_at": "ISO-8601" }
  ],
  "validations": { "<qnum>": Validation },
  "contexts": [ Context, ... ]          // shared case-study blocks captured on this PDF
}
```

### Context

A shared passage/table/diagram that several questions on the same PDF reference (e.g. a Disease Detectives case study). Ids are only unique **within their own bucket** — every PDF restarts its own counter — so any code that needs to look one up across the whole event namespaces the key as `"<bucket>::<id>"` (done in `api_all_questions`, `build_markdown`'s `_all_contexts()`, and the CSV/JSON/PDF export routes; nothing on disk uses this composite key, it's a read-time convention only).

```jsonc
{
  "id":     "ctx_1",
  "title":  "Avian flu outbreak",   // optional human label; falls back to `id` for display when blank
  "text":   "A passage, table caption, or intro the linked questions reference...",
  "images": [ "filename.png", ... ],
  "pages":  [ 4, 5 ]                 // page span — contexts can span pages just like questions
}
```

A question opts in via its `context_id`; review.html then renders the context above the stem and any edit to the shared text flows through to every linked question. The review page's **"🔗 Group questions"** toolbar toggle lets a user multi-select several questions (via card checkboxes or PDF bounding-box clicks, persisting the selection across page turns) and bulk-assign them to a context in one step — either an existing one or a freshly drag-captured one — instead of linking one question at a time via the per-card dropdown.

### Validation

```jsonc
{
  "status":         "correct" | "incorrect" | "uncertain" | "unavailable",
  "correct_answer": "10V" | null,
  "rationale":      "Ohm's law gives V=IR=2×5=10V.",
  "source":         "Ohm's Law (V=IR)",
  "validated_at":   "ISO-8601",
  "validated_by":   "ai" | "human",    // who set this verdict — see below
  "model":          "claude-haiku-4-5-20251001",
  "text_at_validation":   "... first 300 chars of question text ...",
  "answer_at_validation": "10V",
  "stale":          false              // set true if text/answer changed since
}
```

The Browse page (`templates/browse.html`) can set this two ways, both through `PATCH /event/<slug>/api/q/<bucket>/<num>`'s `validation` field (`review_app.py`'s `api_patch_question`, which validates `status` against `{correct, incorrect, uncertain, unavailable}` and rejects anything else with a 400):
- **AI Validate button** — calls the existing stateless `POST /event/<slug>/api/validate-question` (same Haiku check `review.html` already used), then immediately PATCHes the result with `validated_by: "ai"` so it's persisted right away (unlike `review.html`, which only persists via its page-level Save button).
- **Validation dropdown** — a human sets `status` directly; PATCHes `{"validation": {"status": ..., "validated_by": "human"}}`. Sending `{"validation": null}` clears it outright.

Whichever happens most recently simply overwrites `q["validation"]` server-side — no merge logic, no protection either direction, so a human can always override a stale/wrong AI verdict and a fresh AI Validate run can equally override a human's.

## 5. Annotation replay (`apply_annotations`)

Order matters:

1. Drop questions whose number appears in `deleted`
2. Append `added` questions (skip if number collision)
3. Apply `field_overrides[qnum]` to all questions (pipeline-extracted *and* added) — restricted to a hardcoded whitelist of field names in `apply_annotations` (`build_question_bank.py`): `text, choices, answer, topic, focus, page, extra_pages, context_id, image_descriptions, lastEditedBy, lastEditedDateTime, validation`. Adding a new per-question field that should survive reprocess means adding it to this tuple too, or it silently vanishes on the next Reprocess.
4. Image overrides: strip `detached` and any reassigned images from every question, then attach `assignments[fname] = qnum` to the target
5. Attach `validations[qnum]` to each question; mark `stale: true` if `text_at_validation` or `answer_at_validation` no longer matches

A reprocess thus preserves user effort while still applying new pipeline logic to fields the user never touched.

## 6. Topic taxonomy

Per-event. Each event in `events.py` declares a tuple of `topics` (always ending in `Other / General`) and a `topic_keywords` dict of `{topic: [(phrase, weight), ...]}`. `classify_topic(text)` lowercases the text, sums weights for every phrase that appears, and picks the highest-scoring topic; ties go to insertion order; an empty result falls through to *Other / General*.

Tuning principles that apply to every event:
- **Don't include generic terms** that appear across many topics. For Circuit Lab, that means omitting "voltage"/"current" from *Basic Electrical Concepts*; for Thermodynamics, omitting "heat"/"energy" from *Temperature & Heat*. Otherwise one bucket swallows everything.
- **Use multi-word phrases** when possible (`"ohm's law"`, `"q = mc"`, `"carnot cycle"`) — they're more discriminative than single tokens.
- **Weight 5** for unambiguous names of the topic itself (`"kirchhoff"`, `"carnot"`), **4** for strong-signal multi-word phrases, **3** for moderately specific terms, **2** for weak/common-but-relevant terms.

Custom topics created in the UI (via the "+ New topic..." dropdown option) are stored as `q.topic` strings and surface in future dropdowns without modifying `events.py`.

Current taxonomies (full lists live in `events.py`):

- **Circuit Lab** — 10 buckets: Basic Electrical Concepts · Series & Parallel · Kirchhoff's Laws · Power & Energy · Capacitors · Inductors & Magnetism · AC Circuits & Phasors · Semiconductors & Diodes · Measurement & Lab Skills · Circuit Diagrams & Symbols
- **Thermodynamics** — 11 buckets: Temperature & Heat · Specific Heat & Calorimetry · Thermal Expansion · Heat Transfer · Phase Changes & Latent Heat · Laws of Thermodynamics · Gas Laws & Ideal Gases · Entropy & Free Energy · Heat Engines & Refrigerators · Insulation & Building Science · Lab Skills & Measurement

## 6b. Rotating foci

Some Sci-Oly events change their sub-topic focus each year (Anatomy & Physiology cycles through body systems, Dynamic Planet alternates oceanography/glaciers/tectonics, Water Quality picks freshwater or saltwater, Materials Science assigns a single material family). The pipeline models this with two pieces of state:

- `Event.foci: tuple[str, ...]` — the rotation set declared at registration time. Empty for events without rotation (Circuit Lab, Thermodynamics).
- `Question.focus: str` — which rotation slot this question belongs to. Empty for non-rotating events, or for questions the user hasn't tagged yet.

**Auto-population paths:**

- **scio.ly scrape** uses scio.ly's `"<Event> - <Focus>"` naming convention. When the UI's focus dropdown is set, the scrape rewrites `event_name` to `"Anatomy - Endocrine"` before hitting the API AND passes `focus="Endocrine"` to the scraper so every normalized question gets tagged. `scrape_scioly.split_event_and_focus()` also auto-derives `focus` from any event name that already has the `" - "` suffix, so the same code path handles a directly-typed `"Anatomy - Endocrine"`.
- **PDF extraction** does not auto-tag focus — the year-to-focus mapping isn't encoded in scioly_tests.json and varies by division. Users tag focus per-question via the review UI's focus dropdown; the edit lands in `annotations.field_overrides` and replays on reprocess.

**Markdown rendering** (`build_markdown`):

- If `EVENT.foci` is empty OR no question has a focus set → flat layout: `# Event` → `## Topic` → `### Q*`.
- Else: nested layout: `# Event` → `# Focus` (top-level for jump-from-TOC) → `## Topic` → `### Q*`. Foci are emitted in the order declared on the event; questions with `focus == ""` go into a synthetic `(no focus)` group at the bottom. Question numbering is contiguous across the whole document.

**UI:**

- Sources page shows a Focus dropdown on the scrape form only when `EVENT.foci` is non-empty.
- Review cards show a Focus dropdown only when `STATE.foci` is non-empty or the question already has a focus value. The dropdown has a `(no focus)` option and a `+ New focus…` option that prompts for a new value and adds it to `extraFoci` for the rest of the session.
- The new-event form has a "Rotating foci" input that accepts comma-separated values; persisted to `events_custom.json`.

## 7. Multiple-choice splitting (`split_choices`)

Detects MC options in three formats: `a. text`, `A) text`, `(a) text`. Returns `(stem, choices[])` or `(text, [])` if it doesn't look like MC.

Guards (the regex finds too many false positives without these):

- Letters must be strictly ascending starting from A (with one exception below)
- **Implicit-A inference**: if only `b. ... d. ...` survived OCR, the stem ends at the last `?` or `____` before the first letter, and the implicit A is whatever's between
- Average option length must be < 80 chars (longer means it's a multi-part question with `a) ... b) ...`, not MC)
- Stem must contain an interrogative cue (`?`, `____`, `:`, or a word like *which/what/who/how/calculate*) — without this, what looks like the stem is probably the body of an option whose marker was lost

## 8. Markdown output

`build_markdown(all_questions)` groups by topic in the order listed in §6. Within a topic, questions sharing a `context_id` are clustered together (`_cluster_by_context`, anchored at the earliest member's sort position) instead of rendering wherever each one individually sorts to — the shared Context's text/images render once as a `**Case study[: title]**` block immediately before that cluster's questions, which then render with no divider between them (one divider after the whole cluster). `build_markdown` derives the lookup itself via `_all_contexts()` (reads `state["annotations"][*].contexts`, namespaced `bucket::id`) — callers don't need to pass anything, just make sure each question dict carries `_bucket` (both call sites — the CLI's `main()` and `review_app.py`'s `api_regenerate` — tag it before calling). The PDF/CSV/JSON exports (`review_app.py`'s `_export_pdf`/`api_export`) do the same clustering for their own outputs; CSV instead adds a plain `context_id` column (no natural place for a shared passage in flat rows) and JSON wraps the question array as `{"questions": [...], "_contexts": {...}}` instead of a bare array. Each question renders as:

```
### Q<n>
*<source> · #<number>*

<stem>

- **A.** ...
- **B.** ...

![Figure](images/...)

**Answer:** ...

> ✓ **Verified: Correct**  <rationale>  *Source: <source>*
```

Math markers (`$...$`, `$$...$$`) pass through verbatim — GitHub-flavored markdown renders them. Validation blockquote omitted when no verdict exists.

## 9. Flask review UI

State store and pipeline are the same. The UI is just a graphical front-end that produces annotations and generated questions.

### Pages

- `GET /` — event landing page. Coaches see every registered event; volunteers only the ones a coach assigned them (filtered in `index()`, `review_app.py`).
- `GET/POST /login`, `POST /logout` — session login/logout (see Authentication & roles below).
- `GET /admin/users` — coach-only user management page.
- `GET /event/<slug>/` — sortable PDF list for one event. Includes an upload-test-PDF form (test + optional key).
- `GET /event/<slug>/review/<pdf>` — single-PDF review (PDF page + question cards). Accepts `?q=<number>` to focus a specific question on load (used by the "Open source ↗" link from the browse page).
- `GET /event/<slug>/sources` — source-material management + LLM question generation
- `GET /event/<slug>/browse` — event-wide question browser: every question across every bucket on one filterable/sortable page. Filters: full-text search, topic, focus, source, validation status, MCQ-vs-FRQ, has-image. Sorts: Q#, topic, focus, source, validation, length, bucket-then-Q#. Each card links back to the source PDF's review page with that Q pre-focused.
- `GET /event/<slug>/quiz` — client-side practice quiz: configure count/topic/bucket/type/timer/shuffle, fetch the pool from `/api/all-questions`, then step through one question at a time. The **"Keep case studies together"** checkbox (default off, so existing behavior is unchanged unless opted into) clusters the filtered pool by `_context_key` before picking — a picked group's siblings are pulled in from the full pool (even ones the active filters would've excluded, as long as they still have an answer) and shuffled as a unit, so a case study is never split across included/excluded. The question screen shows the shared passage in a banner above the stem, with a "question N of M" indicator, whenever the current question carries a `_context_key`.

Every API route is namespaced under `/event/<slug>/`. Each handler calls `_select_event(slug)` first, which mutates the `build_question_bank` module globals (`BASE_DIR`, `IMAGE_DIR`, `OUT_MD`, `STATE_FILE`, `TOPICS`, `EVENT`) so the rest of the request runs in the right context, and — since B#1 — checks the logged-in user's event access. `bqb.set_event` writes to a `ContextVar`, so this is correct per-request/per-thread under a multi-threaded WSGI server; the only piece that still requires a single worker *process* is the per-event state lock (`build_question_bank.py`'s `threading.RLock` dict — see Deploying in the README).

### Authentication & roles

`auth.py` holds two roles — `coach` (implicit access to every event, plus user/textbook management) and `volunteer` (access only to their assigned `events` list) — backed by `auth_users.json` at the repo root (same flat-JSON, atomic-write pattern as `events.py`'s `events_custom.json`; gitignored, never committed). Passwords are hashed with `werkzeug.security.generate_password_hash`.

- `app.before_request` redirects any request without a valid session to `/login`, except `/login` itself, `/static/*`, and the favicon.
- `_select_event(slug)` (`review_app.py`) is the single chokepoint for per-event access: every `/event/<slug>/...` route calls it first, so adding the access check there (instead of decorating ~80 routes individually) covers all of them. Coaches pass unconditionally; volunteers get `403` if `slug` isn't in their assigned list.
- `@coach_required` gates routes that aren't event-scoped: event create/edit/delete (`/api/events*`), shared-textbook write routes (`/api/textbooks/upload`, `/<id>/detect`, `/<id>/chapters` — reads stay open to volunteers), and the `/admin/users*` routes below.
- `g.user` (set in `before_request`) and the `current_user` Jinja global (set via `@app.context_processor`) make the logged-in user available to every route handler and every template without threading it through each `render_template` call.

| Endpoint | Purpose |
|---|---|
| `GET /admin/users` | Coach-only. Lists every account with role + assigned events. |
| `POST /admin/users` | Coach-only. Body: `{username, password, role, events}`. |
| `PATCH /admin/users/<username>` | Coach-only. Body: `{role?, events?}` — partial update. |
| `DELETE /admin/users/<username>` | Coach-only. Refuses to delete the caller's own account. |

First-time bootstrap: `python auth.py --create-coach` (interactive CLI) creates the first account directly, since a fresh `auth_users.json` has no coach who could otherwise use the admin UI.

### API — review

| Endpoint | Purpose |
|---|---|
| `GET /event/<slug>/api/all-questions` | Flat list of every question in the bank with bucket provenance + computed flags (`_bucket`, `_synthetic_bucket`, `_has_image`, `_is_mcq`, `_validation_status`, `_edited_at`, and `_context_key` — `bucket::context_id`, only present when the question is linked to one) and pre-aggregated stats baskets, plus a top-level `contexts` map (`_context_key` → Context). Powers the browse and quiz pages. |
| `PATCH /event/<slug>/api/q/<bucket>/<num>` | Single-question edit (text/topic/focus/answer/choices/image_descriptions/validation) for Browse's always-editable cards — every field autosaves independently, no page-level Save. Records the change as a `field_overrides[<num>]` annotation so reprocess respects it, and server-stamps `lastEditedBy`/`lastEditedDateTime` from the session on every call that actually changes something (see §4's Question/Validation schemas). |
| `DELETE /event/<slug>/api/q/<bucket>/<num>` | Remove a single question from a bucket; records `<num>` in `annotations[<bucket>].deleted` so reprocess respects it. |
| `GET /event/<slug>/api/export.csv` | Whole bank as CSV (one row per question, columns: number/topic/focus/text/answer/choices/source/year/division/bucket/context_id/validation_status/rationale). |
| `GET /event/<slug>/api/export.json` | Whole bank as JSON: `{"questions": [...], "_contexts": {"bucket::id": Context, ...}}`. |
| `GET /event/<slug>/api/export.pdf` | Printable PDF (reportlab), grouped by topic with an answer key page. Case-study clusters render their shared passage once (a highlighted box) before the cluster's questions — same `_cluster_by_context` grouping as the markdown output (§8). Requires `pip install reportlab`. |
| `GET /event/<slug>/api/export.apkg` | Anki deck (genanki), one card per question — flashcard format has no slot for a shared passage, so context grouping doesn't apply here. Requires `pip install genanki`. |
| `GET /event/<slug>/api/pdf/<pdf>/outline` | PyMuPDF-extracted PDF outline (`doc.get_toc()`) for the review-page outline drawer. Returns `[{level, title, page}]`. |
| `GET /event/<slug>/api/pdf/<pdf>/page/<n>/qboxes` | Per-page bounding boxes for every question whose text starts on this page. Computed on demand from PyMuPDF text blocks: each `Q_START` match anchors a box that spans down to the next anchor (or page bottom), widened to the leftmost/rightmost edges of the text blocks within that vertical slice. Returns `{boxes: [{number, x0, y0, x1, y1}], page_height_pt}` in PDF points; the frontend scales to image pixels using the same 120 DPI factor as region-extract. |
| `POST /event/<slug>/api/generate-similar` | Seed-based question generation. Body: `{seed_text, seed_topic, n}`. Wraps `qgen.generate_questions` with the seed as both source material and "do not duplicate" anchor. |

### Region-coordinate convention

The frontend sends a region in **natural image pixels** at the requested DPI (default 120). The backend converts to PDF points:

```python
f = 72.0 / dpi
pdf_rect = fitz.Rect(x*f, y*f, (x+w)*f, (y+h)*f)
text = page.get_text("text", clip=pdf_rect)
```

Math extraction re-renders that same `pdf_rect` at 240 DPI for vision clarity before sending to Haiku.

### Question bounding boxes (`/qboxes`)

Derived on-demand from PyMuPDF text blocks rather than stored at extraction time, so this works on any PDF in the bank without rerunning the pipeline.

Algorithm:
1. `page.get_text("blocks")` returns `(x0, y0, x1, y1, text, block_no, type)` tuples.
2. Filter to text blocks (type 0) and sort by `y0`.
3. For each block, inspect the first non-blank line — if `Q_START` matches, record an anchor `(y0, qnum, block)`.
4. For each consecutive pair of anchors, compute the bbox spanning from `anchor[i].y0` to `anchor[i+1].y0` (or page height for the last one), widened to the leftmost/rightmost edges of every text block inside that vertical slice.
5. Return PDF-point coords; the frontend scales to displayed image pixels.

UI behavior:
- Boxes render as soft-green overlays after the page image loads.
- **Focused** (the q whose `_id` matches `focusedId`): orange box.
- **Unmatched** (the box's qnum doesn't appear in `STATE.questions`): dashed red box — this is the "the algorithm thinks there's a Q here but the parser dropped it" diagnostic.
- Click a box → jumps to that question's card and focuses it.
- Toggle on/off via the ▢ button in the zoom controls; preference persists in `localStorage` (default ON).
- Key PDF mode hides bboxes (key PDFs aren't question-numbered the same way).

### Status-bar lifecycle

`setStatus(html, kind)` writes into the header's status span and schedules a clear after 4.5s. Earlier versions cleared only when `el.innerHTML === html`, but **browsers normalise attribute quoting when reading back `innerHTML`** (e.g. `'spinner'` → `"spinner"`), so any status containing inline HTML failed the equality check and the message — and any embedded spinner — never cleared. Current implementation uses a monotonically incremented `_statusSeq` token: each call captures its sequence, and the timeout only clears when no newer call has run since. This means the spinner stops as soon as the work finishes, regardless of what shape the status string takes.

Handler convention: every async UI handler that shows a "Working…" status must also write a definitive terminal status — success, cancel, no-op, or error — on every code path, so the user always knows the operation ended. Disable the triggering button via `try/finally` to prevent re-entry while a long-running call is in flight.

### Add-question-from-region UX

When the user clicks **+ Add question from region** and drags a rectangle, the frontend:

1. Sends the region to `extract-region` with `parse_choices: true`
2. Receives `{text, stem, choices}`
3. Auto-assigns the next available numeric Q# — `max(existing.number) + 1` — **no prompt**
4. If `choices` is non-empty, uses `stem` for the question text and populates the choices list; otherwise uses `text` as the stem with an empty choices list

This means dragging a region that covers a stem plus its `A. ... B. ... C. ...` options produces a fully-structured question card in one drag.

## 10. Vision prompts

Stable prompts kept in `build_question_bank.py`:

- **Image-to-question association** — gives Haiku the rendered page, the question numbers that *start* on it, and the count of figures. Asks for `{"<fig_idx>": "<qnum>" | "PREV"}`. The `PREV` escape hatch covers figures that illustrate a question started on the previous page.
- **Page OCR** — for fully image-based PDFs. Asks for `{"questions":[{"number","text","has_figure"}], "answers":[...]}`. Empty `{}` for cover/instruction pages.
- **Key OCR** — for image-based key PDFs. Asks for `{"1":"B","2":"14 Ω", ...}`.
- **Math → LaTeX** — sends a high-DPI crop. Asks for the LaTeX with no delimiters; reply `NONE` if no math.
- **Answer validation** — sends question + choices + recorded answer. Asks for a JSON verdict with status, correct answer, rationale, primary source.

All vision calls go through `_vision_call(b64, prompt, max_tokens)` and are cacheable: callers store results in `state.vision[<pdf>][<key>]` and check the cache before paying again.

## 11b. scioly.org bot bypass (`download_event.py`)

scioly.org uses **Anubis** JavaScript proof-of-work bot protection. Plain `requests` always gets back HTML challenge pages.

The downloader handles this transparently:

1. Issues GETs with a Chrome User-Agent and any cookies from `.scioly_cookies.json`.
2. If a response is `text/html` containing "not a bot" or "anubis", that's the challenge.
3. **First time only** in the run: launches headed Chromium via Playwright, navigates to `https://scioly.org/tests/`, waits up to 60s for `document.title` to no longer contain "not a bot" (signals PoW completion), grabs `context.cookies()`, and saves them to `.scioly_cookies.json`.
4. Re-applies the cookies to the requests session and retries the same URL.

Subsequent runs reuse the saved cookies until they expire. `--reauth` discards them and re-runs the bypass. `--no-bypass-bot` skips Playwright entirely and just fails on a challenge.

Prerequisites: `pip install playwright` and `playwright install chromium` once.

Detection helper: `_looks_like_challenge(response)` checks `content-type` + a substring scan of the first 600 chars.

## 11c. Source materials (`texts.py`)

Each event has a `<event>/texts/` directory holding reference material the LLM can draw on. Three file kinds:

- `wiki.md` — fetched from `event.wiki_url` (scioly.org/wiki/index.php/<Page>)
- `<name>.pdf` — user-supplied PDF (textbook chapter, study guide, notes)
- `<name>.md` — derived markdown (via `pdf_to_markdown`) **or** standalone reference text

`list_sources(event)` returns one entry per source. A PDF + sibling `.md` are merged into one row; the markdown is shown as derived metadata.

`scrape_wiki(event, cookies)`:
- Builds the URL from `event.wiki_url` (default: `name.replace(' ', '_')`; can be overridden via `Event.wiki_page`)
- Uses the saved Anubis cookies if provided
- Extracts `div.mw-parser-output` with BeautifulSoup, strips `mw-editsection`/`noprint`/`toc`/navbox noise
- Converts to markdown via `markdownify(heading_style="ATX")`
- Saves with a small `# <event> — Wiki` header to `<event>/texts/wiki.md`

`pdf_to_markdown(pdf_path)`:
- PyMuPDF text-only extraction, page-by-page
- Writes `## Page N\n\n<text>` blocks to a sibling `.md`

`read_source_text(event, name)`:
- Resolves `foo.pdf` → `foo.md` (errors if not yet processed)
- Resolves `bar.md` → `bar.md`

### Shared textbooks (`textbooks/`, top level — not under any one event)

Textbooks differ from per-event sources in two ways: they're uploaded once and usable from every event, and they're split by chapter so generation targets one chapter's pages rather than the whole book. Storage layout: `textbooks/<id>.pdf` + `<id>.md` (built via the unmodified `pdf_to_markdown` above) + `<id>.chapters.json`.

`detect_chapters(pdf_path, md_text)` — two strategies tried in order:
1. The PDF's own embedded outline (`fitz.open(pdf_path).get_toc()`). When present, the top-level entries are taken as-is (`{title, start_page: page}`) — most reliable when it exists.
2. A regex scan of `md_text` for heading lines matching `^(?:chapter|unit|section)\s+(\d{1,3}|[ivxlcdm]+|[a-z]+)\b` (case-insensitive, start-of-line only so mid-sentence mentions don't match), tracking the most recent `## Page N` marker `pdf_to_markdown` already emits to assign each match's `start_page`.

If neither strategy finds anything, returns `{chapters: [], source: "manual", needs_manual_chapters: true}` so the caller can fall back to manual entry. Otherwise each chapter's `end_page` is filled in from the next chapter's `start_page - 1` (last chapter runs to the PDF's final page).

The manual-entry route (`POST /api/textbooks/<id>/chapters`) applies the identical cascading-`end_page` rule when a chapter's `end_page` is omitted, so the two code paths produce consistent boundaries.

## 11d.1. scio.ly scraper (`scrape_scioly.py`)

scio.ly exposes a public, no-auth REST endpoint that returns Science Olympiad practice questions:

```
GET https://www.scio.ly/api/questions?event=<Display>&question_type=mcq|frq&limit=N&division=B|C
```

Returns `{"success": true, "data": [{ id, question, options, answers, subtopics, tournament, division, difficulty, base52 }, ...]}`. `answers` is a list of indices into `options` for MCQ; for FRQ it holds the canonical text.

The module exposes:
- `fetch_batch(event_name, question_type, limit, division)` — one HTTP call, returns raw records
- `_normalize(raw)` — converts to canonical Question shape: maps `answers: [idx]` → letter "A"/"B"/…; resolves `subtopics[0]` to the event's topic taxonomy when it matches, else falls back to `classify_topic`; sets `source: "scio.ly · <tournament>"`; carries through `_scioly_id` and `_scioly_difficulty` as provenance
- `scrape_questions(event_name, count, types, division, existing_scioly_ids)` — round-robins over types, rate-limits at 0.5 s between API calls (`REQUEST_DELAY`), dedups against the provided UUID set

**Things scio.ly questions notably lack:** images (the `images` list is always `[]`), and many questions are partial — context-stripped or with answers that depend on missing background. The Haiku validator catches these as `uncertain`/`incorrect` so the user can drop them in bulk via the UI's quick-filter buttons.

**Two-layer dedup**:

1. **Exact (UUID)** — each scraped question's UUID is stored in `q._scioly_id`. The scrape endpoint collects every UUID already in `_scioly_<slug>.pdf` and asks the scraper to skip them — so hitting Scrape repeatedly walks deeper into scio.ly's archive instead of returning the same top results. Counted under `skipped_id_dups`.
2. **Fuzzy (text)** — the endpoint also gathers every question from every bucket (PDF-extracted, LLM-generated, prior scio.ly scrapes) into a single `existing_questions` list and passes it to `scrape_questions(..., existing_questions=...)`. The scraper calls `qgen.is_duplicate(candidate, existing_questions + already_kept_in_this_batch, threshold=0.4)` on each normalized question. Hits are dropped into `rejected_text_dups[]` with `{text, matched_q_number, matched_q_source, matched_q_text, _scioly_id}` so the UI can render the side-by-side comparison.

To make the fuzzy layer work despite scio.ly returning random samples per call, the scraper requests `min(per_type * 3 + 10, 50)` records per type — over-fetching by ~3× so dedup filtering still leaves enough to satisfy `count`.

## 11d. Question generation (`qgen.py`)

`generate_questions(source_text, n, types, existing_questions, max_chunks=1)` returns:

```jsonc
{
  "candidates":     [ Candidate, ... ],
  "rejected":       [ {"reason": "duplicate" | "too short", "matched": "<qnum>" | null, "text": "..."} ],
  "warnings":       [ "chunk 1: ...", ... ],     // non-fatal issues per chunk
  "model":          "claude-haiku-4-5-20251001",
  "chunks_used":    1,
  "raw_count":      5,
  "out_max_tokens": 8850,                          // budget actually requested
  "generated_at":   "ISO-8601",
  // If the LLM fails outright, these appear instead of candidates/rejected:
  "error":          "LLM response was not valid JSON.",
  "stop_reason":    "max_tokens",                  // from Anthropic API
  "raw_response":   "<first 1500 chars of raw LLM output>"
}
```

**Token budgeting.** `out_max_tokens = min(16000, 600 + 550 * n * chunks)`. Each generated question with rationale + source snippet runs ~500 tokens; a few-hundred-token overhead covers the JSON envelope. Without this scaling, large `n` requests would silently get truncated mid-JSON, producing 0 candidates with no error.

**Failure surfacing.** When `_parse_json` returns None on the first (or only) chunk, the function returns `error` + `stop_reason` + the first 1500 chars of `raw_response` so the UI can show the user what came back. Per-chunk failures on a multi-chunk run accumulate into `warnings` and the run continues.

**Frontend UX during generation.** The Generate button is disabled and replaced with "Generating…" while in-flight; a progress panel shows the elapsed-time counter with phase hints ("Sending request…", "Waiting for Haiku…", "Still drafting…", "Taking longer than usual — you can Cancel if needed"). A Cancel button uses `AbortController` to abort the fetch. HTTP errors and non-JSON server responses are caught and displayed in the status bar.

Pipeline:

1. **Chunk** source via `chunk_text(text, max_chars=9000)` on paragraph boundaries.
2. For each chunk (up to `max_chunks`, default 1), build a structured prompt:
   - Identifies the event by name
   - Spells out per-type expectations (mc: 4 choices A-D with plausible distractors; short: 1-2 sentence answers; numerical: answers with units + equation in rationale)
   - Lists the event's `topics` and asks Haiku to pick one per question
   - Shows up to 30 existing question stems as "do not duplicate"
   - Demands JSON-only output with schema: `{questions:[{type, topic, text, choices, answer, rationale, source_snippet}]}`
3. **Clean** each candidate: `_strip_points` on text/answer/choices, re-letter choices so they start at A, classify topic from text if Haiku picked something off-list.
4. **Dedup**: `is_duplicate(cand, existing, threshold=0.4)` uses Jaccard similarity on word-3-grams of the stem. Auto-rejects with `reason: "duplicate"` + the matched question number. Candidates are also deduped against earlier candidates in the same batch.

`candidate_to_question(cand, number, source_label)` converts a kept candidate to the canonical Question shape used by the rest of the bank, populating the `validation` field with the LLM rationale, source snippet, and `generated: true` so generated questions are distinguishable in the markdown and UI. Generated questions are stored under a synthetic state key `_generated_<event>.pdf` so they survive reprocess and show up alongside real ones.

## 13. Known limitations

- **Multi-section tests** lose most automatic answer matching (e.g. tests with separate "History", "Capacitors", "Op-Amps" sections each restarting at 1). The pipeline detects this case (any suffixed question number) and skips text-based key matching to avoid wrong answers. The reviewer is the recommended path for these.
- **OCR-only PDFs** with no embedded text rely entirely on Haiku; quality varies. Three known PDFs (`cookiepie1`, `ivanioc`, plus one mis-encoded filename) still yield 0 questions because they're structurally empty.
- **Topic classification is keyword-based**. It works well on the vocabulary captured in each event's `topic_keywords`; novel phrasings fall to "Other / General". Custom topics from the UI live on individual questions but don't update the keyword scorer — so the next pipeline run on a fresh PDF won't auto-classify into a user-invented topic.
- **Validation accuracy is bounded by Haiku 4.5**. Treat verdicts as suggestions for a human to confirm, not gospel.
- **scioly.org bot protection** is Anubis JavaScript proof-of-work. `download_event.py` and the wiki scraper auto-handle it via Playwright (see §11b). Requires `playwright install chromium` once. Cookies expire after a few hours; `--reauth` (downloader) or simply re-running scrape-wiki after expiry forces a fresh challenge.
- **Generated questions are not validated against authoritative sources** — Haiku's rationale + source snippet are the LLM's claim, not a check. Treat them as draft material: use the answer-validation button (§9) to cross-check generated questions before relying on them.
- **Generation prompts only use the first chunk** of a source by default (`max_chunks=1`). For a long textbook this skews coverage to the early sections. Repeat the Generate action or pre-chunk the source by hand to spread coverage.

## 14. Security hardening

### No-permanent-deletion data model

Every destructive action the app exposes — including to coaches — is now a reversible flag flip or a pre-action snapshot, not a real delete. The only true deletion left is an operator-run CLI, never a web route.

**`archive.py`** — snapshot/restore for destructive reprocesses, plus the operator-only purge:
- `snapshot_pdf_state(event, pdfname, state)` writes `state["annotations"]/["manual"]/["questions"]` for one PDF to `<event_base_dir>/.archive/<pdfname>/<ISO-timestamp>.json` (atomic tempfile+`os.replace`, same pattern as `build_question_bank.py`'s `_save_state`). No retention limit — small JSON, not PDFs.
- `list_snapshots`/`load_snapshot` back the review-page UI. `load_snapshot` validates the filename against a strict timestamp regex before touching disk — rejects path traversal outright.
- `python archive.py --purge-snapshots-older-than-days N` is the only thing that actually deletes a snapshot file. Never called from `review_app.py`.

**Reprocess hook** (`review_app.py`'s `api_reprocess`): when `discard_annotations` is set and the PDF actually has annotations/manual-edits/questions to lose, `archive.snapshot_pdf_state(...)` runs *before* the wipe. `GET .../snapshots` and `POST .../restore-snapshot` (same `_select_event` access gate as every other PDF route — restoring is protective, not destructive) round out the UI in `templates/review.html`'s Reprocess dropdown.

**Event archive** (`events.py`): `Event.archived: bool = False`, threaded through `_event_to_dict`/`_dict_to_event` same as every other field. `archive_custom_event`/`unarchive_custom_event` replace the old `remove_custom_event` — built-ins still can't be archived (same `is_builtin()` guard as edit/delete). `index()` (`review_app.py`) excludes archived events from the normal landing-page list and surfaces them to coaches in a separate "Show archived" section; `_select_event` 404s any direct access to an archived event until it's unarchived.

**User disable** (`auth.py`): `User.disabled: bool = False`, same threading through `_load`/`_save`. `disable_user`/`enable_user` wrap `update_user`. `verify_login` rejects disabled accounts outright; `_require_login`'s `before_request` (`review_app.py`) treats a disabled user exactly like a deleted one — clears the session and redirects to `/login` on their very next request, so a disable takes effect immediately even for an already-logged-in session. `auth.delete_user` (true removal) still exists as a primitive but is only ever meant to be called from an operator shell, not from any route.

### Upload & request hardening

**`pdf_safety.py`** — defensive PDF opening shared by `build_question_bank.py`'s `process_pair` and `texts.py`'s `pdf_to_markdown`/`detect_chapters`:
- `looks_like_pdf(path)` checks for the `%PDF-` magic header — cheap, called both inline in the three upload routes (immediate rejection, deletes the bad upload) and inside `open_pdf_safely`.
- `open_pdf_safely(path)` runs `fitz.open()` in a `ThreadPoolExecutor` with a 30s timeout (deliberately not a `with` block — `Executor.__exit__` calls `shutdown(wait=True)`, which would block on a genuinely hung open; `shutdown(wait=False)` lets the request return promptly and leaks the stuck worker thread instead, an acceptable tradeoff at this app's request volume) and rejects anything over `MAX_PAGES = 2000` right after open (cheap metadata read) and before any caller loops over every page.

**SVG sanitization**: `_sanitize_svg` (`review_app.py`) strips `<script>` elements and `on*=` attributes from an uploaded SVG question image before it's written to disk. Not a full sanitizer (doesn't touch `<foreignObject>` or external refs) — defense-in-depth on top of the app only ever rendering these via `<img>` tags, which don't execute embedded scripts.

**Path containment**: `_safe_join(base_dir, user_supplied_name)` (`review_app.py`) runs `secure_filename()` then verifies the resolved path `.is_relative_to(base_dir.resolve())` before returning it, `abort(400)` otherwise. Applied to `serve_image`, `api_process_source`, `api_source_raw` — the three routes that build a filesystem path from a single dynamic URL segment.

**CSRF**: double-submit cookie. `/login` sets a non-HttpOnly `csrf_token` cookie alongside the session cookie. `_check_csrf` (`before_request`, registered after `_require_login`) rejects any `POST`/`PUT`/`PATCH`/`DELETE` whose `X-CSRF-Token` header doesn't match the cookie (`secrets.compare_digest`), except `login`/`logout` (native `<form>` POSTs, can't attach a custom header — both are auth-flow routes, not data mutations). The frontend's existing `window.fetch` monkey-patch (the one that auto-attaches `X-LLM-Keys`, see §9) gained a sibling patch that reads the cookie and attaches `X-CSRF-Token` to every non-GET/HEAD request — no per-call-site changes needed anywhere in the templates.

**Login rate limiting**: `_login_attempts` (`review_app.py`), an in-process `dict[ip, list[float]]` guarded by a `threading.Lock`. 5 failures within a 15-minute sliding window locks that IP out (`429`) until the window clears; a successful login clears its history. Deliberately keyed on `request.remote_addr`, never `X-Forwarded-For` — this app runs no trusted-proxy config, so honoring a client-supplied header would let an attacker spoof a fresh IP per request and bypass the limiter entirely. Behind Caddy without further config every visitor shares Caddy's address (rate-limits the whole app together) — fails safe rather than spoofable. Resets on process restart.

**Fail-fast startup config**: two checks, both in `review_app.py`. (1) Module level: if `SESSION_COOKIE_SECURE=true` (the operator's own signal this is a real HTTPS deployment) but `FLASK_SECRET_KEY` isn't set, `sys.exit` before the app object finishes configuring — catches the gunicorn/production path. (2) In `main()`: if `--host` isn't loopback (`127.0.0.1`/`localhost`/`::1`) and `FLASK_SECRET_KEY` isn't set, `sys.exit` before `app.run()` — catches the `python review_app.py` dev-server path exposed to a real interface. Neither check fires for default local-dev usage (loopback host, no env vars set), which keeps working exactly as before.

**Subpath mounting**: `_PrefixMiddleware` (`review_app.py`) wraps `app.wsgi_app` when `APPLICATION_ROOT` is set in the environment — the standard Werkzeug pattern for an app sitting behind a reverse proxy that mounts it under a path prefix (e.g. `/testbank/ncms`) instead of the domain root. It strips the prefix from `PATH_INFO` into `SCRIPT_NAME`, so `url_for()` and `request.script_root` produce correctly-prefixed URLs everywhere, including the redirect target built from `request.script_root + request.path` in `_require_login`. Server-rendered template links use `url_for(...)`/`request.script_root` directly; client-side JS has no equivalent, so `_user_badge.html` injects a `const APP_ROOT = "{{ request.script_root }}";` global that every page's inline `<script>` block prefixes its `fetch()` URLs with. Unset (the local-dev default), the middleware is a no-op and `APP_ROOT` is `""`. The reverse proxy must forward the full path **unstripped** — Caddy's `handle /testbank/ncms* { ... }` (not `handle_path`) — since the middleware does its own stripping.

**Cookie path isolation**: `app.config["APPLICATION_ROOT"]` and `app.config["SESSION_COOKIE_PATH"]` are set from the same `APPLICATION_ROOT` env var (a separate Flask config key from the plain module-level variable `_PrefixMiddleware` uses — both must be set), and the `csrf_token` cookie's `set_cookie()` call passes `path=app.config["SESSION_COOKIE_PATH"]`. This matters once more than one instance is mounted under different prefixes on the same domain (see "Running multiple independent instances" below) — without it, every instance's session/CSRF cookies default to `Path=/` and the browser sends each instance's cookies to every other instance's routes too. A different `FLASK_SECRET_KEY` per instance already makes this safe (a foreign session cookie just fails signature validation and is treated as logged-out), but it's wasteful and was worth closing properly.

**Running multiple independent instances** (e.g. two schools on one server): each instance is a fully separate code+data directory tree (`REPO_ROOT` is `Path(__file__).parent` throughout `auth.py`/`events.py`/`archive.py`, with no override, so two instances sharing a directory would share `auth_users.json`, event data, everything) — own system user, own `.env`/`APPLICATION_ROOT`, own systemd unit bound to its own `127.0.0.1:<port>`, one more `handle` block in the same Caddyfile. The Python venv is independent of `REPO_ROOT` and *can* be shared across instances. See README.md's "Running multiple independent instances" and "Updating code across multiple instances from GitHub" sections for the full setup, and `deploy/update-from-github.sh`/`deploy/_apply-update.sh` for the redeploy mechanism (split into a low-privilege fetch step and a root-only, non-parameterized, narrowly-sudoer'd validate+sync+restart step). See §15 for the backup mechanism's rationale, and README.md's "Production deployment (current state)" for the concrete accounts/paths/ports actually in use.

## 15. Backup mechanism

README.md's "Backups" section covers *what* runs (`deploy/backup-extracted-data.sh`, `deploy/backup-bulk-data.sh`, `deploy/qbank-commit-message.py`); this covers *why* it's split the way it is.

**Why two storage technologies instead of one**: the data splits into two kinds with genuinely different recovery needs. Extracted/derived data (`.qbank_state.json`, `question_bank.md`, `events_custom.json`) is text/JSON where line-level diffs and "who changed what, when" matter — exactly what git is for. Bulk source data (PDFs, `texts/`, `textbooks/`) is binary where diffing byte-for-byte is meaningless but durable, cheap, recoverable-if-deleted storage matters — S3 via `restic` (client-side encrypted, deduplicated, with its own retention/pruning), not git. A single private GitHub repo could technically hold both (with Git LFS for the binaries), but that fights git's design and risks GitHub LFS's free-tier bandwidth/storage caps as libraries grow — two tools, each doing the job it's good at.

**Why extracted images ended up in *both* places**: originally scoped to S3-only under the "binary → S3" rule above. Moved into the git databank too once actual volume was confirmed small (12MB/415 files across all of NCMS's events at the time) — plain git has no problem at that scale, no LFS needed. This is a per-deployment judgment call, not a fixed architectural rule; re-check the assumption (`du -sh */images`) if any event's image set grows into the hundreds of MB, since at that point the GitHub-repo copy stops being a low-cost convenience and starts being a real git-performance and LFS-cost concern.

**Why the commit-message diff is narrow**: `qbank-commit-message.py` only looks at `lastEditedBy`/`lastEditedDateTime` stamps (§4's Question schema) when building a human-readable commit message. Pipeline-driven changes — a fresh PDF processed, new images extracted — don't carry those stamps (only `api_patch_question`, a human edit through the Browse page, sets them), so a backup run that's purely pipeline output falls through to a generic timestamp commit message rather than a fabricated "edited by" claim. This is by design: the script's job is surfacing genuine *human* edits distinctly from pipeline noise, not summarizing every byte that changed — `git log -p` is still there for the full picture on a generic-message commit.

**Why backup-destination credentials live outside the app's own `.env`**: same reasoning as the rest of this section's secret-isolation work (cookie path isolation, fail-fast startup config) — `/opt/qbank/backup/.env` is read only by the backup scripts (cron jobs), never by `review_app.py`/gunicorn, so a future app-level vulnerability can't also leak GitHub/AWS/restic credentials it never had access to in the first place.

**Why `auth_users.json`/`.env` are excluded from both backup pipelines**: these are the one place where "automatic and unattended" is the wrong tradeoff. Everywhere else, more frequent backup is strictly safer — for password hashes and API keys, an unattended pipeline is one more place those secrets could leak from. Handled manually instead (e.g. an `age`-encrypted snapshot to a key only the operator holds), traded deliberately for that smaller blast radius even though it means these two files only ever get backed up when someone remembers to do it.

**The restic-password failure mode, stated plainly**: restic's encryption has no recovery mechanism by design — encrypted data with no surviving password is equivalent to having no backup at all, even though the (now-permanently-opaque) bytes are still sitting in S3. This is the one secret in the whole design where "lost" and "never existed" have identical practical consequences.

## 16. Server-admin app

`admin_app.py` is a separate Flask app (own process, own system user `qbank-admin`, own sudoers grants) at `/testbank/admin` for stop/start/restart/update/rollback/console operations on the other instances — see README.md's "Admin app" section for what it does. This section covers the two design decisions that depart from patterns set elsewhere in this codebase.

**Why a separate app instead of a route in `review_app.py`**: every other privilege boundary in this codebase (per-instance system users, per-instance `.env`s, the fetch/apply split in the GitHub-update mechanism) exists to keep one compromised or buggy surface from reaching another's resources. A route inside `review_app.py` that can stop/start systemd units would mean the question-bank review process — the one surface handling arbitrary uploaded PDFs and LLM output — also needs root-adjacent privileges. Splitting it into its own process means a bug or compromise in either app doesn't inherit the other's privileges for free.

**Why `qbank-rollback.sh` and `qbank-service-ctl.sh` take arguments, unlike `qbank-apply-update.sh`**: the original apply script's safety model (§14, "Running multiple independent instances") is "fixed path, zero arguments, nothing for an attacker or bug to redirect." Rollback and stop/start genuinely need a target (which instance, which backup) — there's no fixed-path equivalent that doesn't also need parameters. Rather than weaken the sudoers grant (sudoers argument matching is itself fragile — a command granted with no arguments specified actually permits *any* arguments unless you explicitly pin an empty argument list), both scripts move the validation into the script itself: every argument is checked against a closed set — the literal instance names in `deploy/instances.conf`, and (for rollback) an actual existing directory name under that instance's backup root — before anything is interpolated into a path or a `systemctl`/`rsync` command. An unrecognized instance name or backup timestamp is rejected outright, with no globbing or partial matching. This is a deliberate, narrower exception to the "no arguments" rule, not an abandonment of it: the script remains the trust boundary, same as before, it just does more validation work to earn that role. `admin_app.py` also re-validates both arguments against the same `instances.conf` before ever invoking `sudo`, as defense-in-depth on top of the scripts' own checks.

**Why the GitHub-fetch step runs as `qbank-deploy`, not `qbank-admin`**: `admin_app.py`'s Update button invokes `sudo -u qbank-deploy bash /opt/qbank-deploy/update-from-github.sh` rather than being granted its own fetch logic. This preserves the existing fetch/apply split exactly — `qbank-admin` never gets root, and never gets `qbank-deploy`'s filesystem access either, only the ability to *run* qbank-deploy's already-narrow script. The nested `sudo` inside that script (to reach the root-only apply step) still resolves against `qbank-deploy`'s own existing grant, unchanged by any of this.

**Why console viewing needs no sudo at all**: `qbank-admin` is a member of the `systemd-journal` group, which grants read-only journal access by default on systemd systems — a strictly lower-privilege mechanism than sudo for a read-only action, kept separate from the write actions (stop/start/restart/update/rollback) that do need the sudoers grants above.
